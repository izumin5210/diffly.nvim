1. Introduction                                           *diffly-introduction*

                                                                       *diffly*
diffly.nvim is a `difit <https://github.com/yoshiko-pg/difit>`-inspired diff
viewer: a file-tree diff viewer (like diffview.nvim) with per-file "viewed"
marks that persist across viewer sessions for the same pull request.

Diffs are always computed from local git: the current branch (or its
checked-out state) is compared against `merge-base(base, HEAD)`. A GitHub
pull request, when one is detected for the current branch, is used only as
metadata: to resolve the base branch and to key the viewed state -- reviewing
someone else's PR still means checking it out first (e.g. `gh pr checkout`).

Diffs display either side-by-side (native Neovim diff mode) or unified
(single column, `<leader>s`/`s` toggles between them -- see
|diffly-keymaps|). Both show the real file -- not a synthetic patch buffer --
so either mode gets the file's own syntax highlighting and LSP; unified
overlays the diff on top of it with extmarks, and deleted lines render as
read-only virtual lines rather than real ones. The buffer is editable in
worktree mode (the default), in both modes alike.

`setup()` is entirely optional; the plugin works with its defaults untouched.
Call it only to override them: >lua
  require("diffly").setup({
    right = "worktree",
    panel = { width = 40 },
  })
<

2. Commands                                                   *diffly-commands*

                                                                      *:Diffly*
:Diffly [{base}]         Open the review UI in a dedicated tabpage, comparing
                        the current branch against {base} (or the resolved
                        default base branch; see |diffly-viewed-state| for how
                        the base is picked). If the UI is already open,
                        focuses its tabpage instead of opening a second one.

:Diffly close            Close the review UI: saves viewed state, closes the
                        dedicated tabpage, and returns to the tabpage `:Diffly`
                        was originally invoked from.

:Diffly toggle           `:Diffly close` if open, otherwise the same as a bare
                        `:Diffly`.

:Diffly refresh          Recompute the diff (same as pressing the panel's
                        `refresh` key, see |diffly-keymaps|). Also runs
                        automatically on `BufWritePost`/`FocusGained`.

:Diffly focus            Focus the panel window (same as
                        |<Plug>(diffly-focus-panel)|, or the `focus_panel` key
                        in |diffly-keymaps|). A no-op with a notice when no
                        review is open.

:Diffly sweep [{name}]   Sweep a `viewed_patterns` group (same as the panel's
                        `S` key): with no {name}, 0 configured groups notify
                        that none are set, exactly 1 sweeps immediately, and
                        2+ prompt via |vim.ui.select()|; {name} (which may
                        contain spaces) sweeps that one group directly,
                        matching an exact name first then a unique prefix.
                        See |diffly-bulk-viewed|.

:Diffly comments         List every comment in the current review in the
                        |quickfix| list -- one item per comment at its
                        last-known line, first body line only, with
                        `[outdated]`/`[base]`/`[@author]` markers. See
                        |diffly-comments|.

:Diffly submit           Post the review's local comment drafts as ONE GitHub
                        review: validates every draft against the PR's real
                        diff (unmappable ones are reported and stay local),
                        prompts for the review event via |vim.ui.select()|,
                        takes an optional summary, then POSTs. Only available
                        for PR-keyed reviews with local HEAD at the PR head.
                        See |diffly-comments|.

:Diffly clean            Prompt to delete the *current* review's persisted
                        viewed state (the key the currently open review uses,
                        or the key that would be resolved if none is open).

:Diffly clean all        Prompt to delete every persisted viewed-state file
                        for every review, across all repos.

Completion offers the subcommands above plus the repo's local branch names,
so `:Diffly <Tab>` can be used either to pick a subcommand or to jump straight
to reviewing against a specific base branch. `:Diffly sweep <Tab>` completes
differently: it offers the current review's configured `viewed_patterns`
group names instead (spaces backslash-escaped), and only when a review is
open on the current tabpage.

3. Configuration                                                *diffly-config*

Pass any subset of these to |diffly.setup()| (called as
`require("diffly").setup({...})`); unset fields keep their default. Full
defaults, exactly as declared in `lua/diffly/config.lua`: >lua
  {
    base = nil,             -- string|nil: base branch override
    right = "worktree",     -- "worktree"|"head"
    include_untracked = true,
    max_file_size = 1024 * 1024, -- bytes; false disables the guard (see below)
    collapse_generated = true, -- collapse generated files (see below)
    auto_advance = true,    -- jump to next un-viewed file after marking
    icons = true,           -- use mini.icons / nvim-web-devicons when present
    viewed_patterns = {},   -- glob pattern GROUPS for bulk-viewed marking (S/sweep)
    panel = { width = 35 },
    keymaps = {
      panel = {
        open = "<CR>",      -- open file diff / toggle fold on a dir row
        toggle_viewed = "v",
        refresh = "R",
        toggle_mode = "s",    -- side-by-side <-> unified
        close = "q",
        fold = "za",
        toggle_hide_viewed = "H", -- hide/show already-viewed rows (display)
        sweep = "S",             -- sweep a viewed_patterns group (prompts if 2+)
        toggle_viewed_subtree = "V", -- bulk-toggle every file under a dir row
      },
      -- applied ONLY in diffly-owned buffers (blob/unified), IN ADDITION to
      -- keymaps.universal
      diff = {
        toggle_viewed = "v",
        toggle_mode = "s",
        focus_panel = "<leader>e",
        close = "q",
        comment_add = "ca",      -- normal: cursor line; visual: the range
        comment_edit = "ce",
        comment_delete = "cd",
        comment_toggle = "ct",   -- collapse/expand inline comments
        comment_toggle_resolved = "cr", -- reveal/hide resolved PR threads
        comment_copy = "cy",     -- AI prompt for the comment at the cursor
        comment_copy_all = "cY", -- AI prompt for every comment
      },
      -- the universal layer: works everywhere (panel, owned diff buffers, AND
      -- real file buffers shown in the viewer); real file buffers get ONLY
      -- this group
      universal = {
        toggle_viewed = "<leader>v",
        toggle_mode = "<leader>s",
        focus_panel = "<leader>e",
        next_file = "]f", -- open the next file (ALL files, unaffected by H)
        prev_file = "[f", -- open the previous file (ditto)
        comment_add = "<leader>ca",
        comment_edit = "<leader>ce",
        comment_delete = "<leader>cd",
        comment_toggle = "<leader>ct",
        comment_toggle_resolved = "<leader>cr",
        comment_copy = "<leader>cy",
        comment_copy_all = "<leader>cY",
      },
    },
  }
<
Notes:
- `base`: overridden per-invocation by `:Diffly {base}`; when both are unset,
  the base is auto-detected (a GitHub PR's base branch when one exists for
  the current branch, otherwise `origin/HEAD`/`main`/`master`).
- `right`: which side of the diff the right-hand pane shows. `"worktree"`
  (default) shows uncommitted edits immediately and lets you edit + `:w`
  straight from the diff; `"head"` compares against the last commit instead.
- `include_untracked`: whether untracked files show up as added ("A") files.
- `max_file_size`: guards against loading a huge text file's full content into
  a diff view; see |diffly-large-files|.
- `collapse_generated`: collapses vendored/lockfile/codegen output behind a
  placeholder, GitHub-parity; see |diffly-generated-files|.
- `icons`: silently does nothing when neither `mini.icons` nor
  `nvim-web-devicons` is installed.
- `viewed_patterns`: glob pattern GROUPS for `S`/`:Diffly sweep`'s bulk-toggle;
  see |diffly-bulk-viewed|. Empty by default -- `S`/`:Diffly sweep` notify that
  nothing is configured until this is set.
- Setting any keymap value to `false` disables that mapping.

4. Keymaps                                                     *diffly-keymaps*

No global keymaps are defined. All of the following are buffer-local and
configurable via `keymaps.universal`/`keymaps.panel`/`keymaps.diff` (see
|diffly-config|).

diffly.nvim uses a two-layer keymap model, modeled on diffview.nvim: a
UNIVERSAL layer of leader-prefixed keys that work identically in every diffly
context -- the panel, diffly-owned diff buffers, and real file buffers shown
in the viewer alike -- plus LOCAL single-key shortcuts that only apply where
the buffer is diffly-owned (so they can never collide with a real file's own,
unrelated keymaps).

Universal (all diffly windows) ~
  <leader>v    Toggle the current file as viewed/un-viewed (auto-advances to
               the next un-viewed file when `auto_advance` is enabled).
  <leader>s    Toggle between side-by-side and unified diff modes.
  <leader>e    Focus the panel (same as `:Diffly focus`).
  ]f           Open the next file (ALL files, not just un-viewed ones).
  [f           Open the previous file (ditto).
  <leader>ca   Add a comment on the cursor line, or the visual selection.
  <leader>ce   Edit the comment under the cursor.
  <leader>cd   Delete the comment under the cursor (confirms first).
  <leader>ct   Collapse/expand inline comment rendering (session-wide).
  <leader>cr   Reveal/hide resolved PR review threads (session-wide).
  <leader>cy   Copy the comment under the cursor as an AI prompt.
  <leader>cY   Copy every comment in the review as one AI prompt.
               (See |diffly-comments| for all of these.)

  Works everywhere: the panel, diffly-owned blob buffers, and real file
  buffers currently shown in the viewer -- worktree-mode side-by-side's
  right-hand window, and worktree-mode unified's single window alike -- the
  one place these keys matter most, since real buffers get no local
  shortcuts at all. There is no universal `close`: closing a real file
  buffer isn't "closing the review". On a real file buffer these maps are
  applied only while it is the one currently open in the view, and are
  removed again the moment the view opens a different file or the review
  closes -- a real file buffer never keeps diffly's keymaps once the viewer
  stops showing it.

  `]f`/`[f` always cycle through every file in the review, regardless of
  the panel's `H` filter (see below) -- the filter only changes what the
  tree *draws*, never what is reachable. Skipping already-viewed files
  while moving forward is what `<leader>v`/`v`'s auto-advance is already
  for.

File panel ~
The row for whichever file is currently open in the diff view is highlighted with the
`DifflyCurrentFile` highlight group (linked to `QuickFixLine` by default; override it like
any other diffly highlight group, e.g. `:hi link DifflyCurrentFile Visual`).

  <CR>    Open the file under the cursor (or toggle a directory's fold).
  v       Toggle the file under the cursor as viewed/un-viewed. When
          `auto_advance` is enabled (the default), the cursor and diff jump
          to the next un-viewed file.
  R       Refresh: recompute the diff against the same base.
  s       Toggle between side-by-side and unified diff modes.
  q       Close the review UI (same as `:Diffly close`).
  za      Toggle the fold of the directory under the cursor (or its parent,
          when the cursor is on a file row) -- mirrors native `za`.
  H       Toggle hiding already-viewed files. Display only: progress counts
          (`3/12 viewed`) stay global, and navigation (`]f`/`[f`, `v`'s
          auto-advance) is entirely unaffected. While active, the progress
          line gains a `(hidden)` suffix, e.g. `3/12 viewed (hidden)`.
          Directories left with no un-viewed files disappear along with
          their contents.
  S       Sweep a `viewed_patterns` group (tri-state; prompts via
          |vim.ui.select()| when 2+ groups are configured; see
          |diffly-bulk-viewed|). Same effect as `:Diffly sweep`.
  V       On a directory row: bulk-toggle every file in that subtree
          (tri-state; see |diffly-bulk-viewed|). On a file row: identical to
          `v`.

diffly diff buffers ~
Diffly-owned buffers only -- the side-by-side blob windows, and unified's own
blob buffers (HEAD mode, or a deleted file's read-only content) -- never
real file buffers, which get only the universal layer above.
  v            Toggle the current file as viewed/un-viewed (same key and
               semantics as the panel's `v`).
  s            Toggle between side-by-side and unified diff modes.
  q            Close the review UI (same as `:Diffly close`/the panel's `q`).
  ca ce cd ct  The comment family, single-key (|diffly-comments|): same
  cr cy cY     actions as the `<leader>c…` universal maps above. Only present
               where there is content to anchor a comment to -- never on
               binary/oversized/generated placeholders, and only in buffers
               showing base content for the base side (the side-by-side left
               window, which is also where deleted lines are commented on).

                                                 *<Plug>(diffly-toggle-viewed)*
                                                    *<Plug>(diffly-toggle-mode)*
                                                   *<Plug>(diffly-focus-panel)*
                                                     *<Plug>(diffly-next-file)*
                                                     *<Plug>(diffly-prev-file)*
Five `<Plug>` mappings are available for binding your own keys, e.g. to
reach these actions from buffers diffly doesn't map by default:
  <Plug>(diffly-toggle-viewed)    Toggle the viewed mark for the file backing
                                 the current buffer, when that buffer belongs
                                 to the currently open review.
  <Plug>(diffly-toggle-mode)      Toggle between side-by-side and unified.
  <Plug>(diffly-focus-panel)      Focus the panel (same as `:Diffly focus`).
  <Plug>(diffly-next-file)        Open the next file (same as `]f`).
  <Plug>(diffly-prev-file)        Open the previous file (same as `[f`).

None has a default binding; map them yourself if wanted: >lua
  vim.keymap.set("n", "<leader>gv", "<Plug>(diffly-toggle-viewed)")
  vim.keymap.set("n", "<leader>gs", "<Plug>(diffly-toggle-mode)")
  vim.keymap.set("n", "<leader>gp", "<Plug>(diffly-focus-panel)")
  vim.keymap.set("n", "]f", "<Plug>(diffly-next-file)")
  vim.keymap.set("n", "[f", "<Plug>(diffly-prev-file)")
<

5. Large files                                             *diffly-large-files*

Images and other binary files never load their content -- both diff modes
always render a placeholder for them, regardless of size. `max_file_size`
(|diffly-config|, 1 MiB by default) extends the same idea to huge TEXT files:
opening one shows a placeholder instead of loading it into a buffer, styled
like the binary one -- >
  file too large (2.3 MiB > 1.0 MiB) -- press L to load
<
with a buffer-local `L` key to load it anyway, for the rest of that view
(side-by-side and unified each track this separately; switching modes or
`:Diffly close` resets it -- it is never persisted).

Only actually loading a file's content into a diff view is guarded:
- Panel counts (status letter, `+`/`-`) come from `git diff --numstat`, which
  stays cheap regardless of file size, so panel rows are entirely unaffected.
- Marking a file viewed (`v`/`<leader>v`/`V`/`S`) never requires loading it
  first -- viewed state is tracked independently of what the view currently
  renders.
- Binary detection always takes precedence: a binary entry never shows size
  text or an `L` key, since there is nothing further to load.

Set `max_file_size = false` to disable the guard entirely.

6. Generated files                                     *diffly-generated-files*

Mirrors GitHub's own "Generated files are not rendered by default" PR-diff
behavior: a file recognized as vendored/lockfile/compiler output keeps its
row, `+`/`-` counts, and manual viewed marking in the panel -- only its diff
body is replaced by a placeholder -- >
  Generated files are not rendered by default -- press L to load
<
with the same buffer-local `L`/per-view force-load mechanics as
|diffly-large-files| above.

Detection ~
Two sources, evaluated lazily when a file is opened:
- github-linguist's own `generated.rb` heuristics -- vendored trees
  (`node_modules/`, `Pods/`, Go's `vendor/`, ...), lockfiles
  (`package-lock.json`, `Cargo.lock`, `pnpm-lock.yaml`, ...), and
  compiler/codegen markers (Go's `// Code generated ...`, protobuf/gRPC/
  Thrift headers, minified JS/CSS, source maps, and many more -- see
  `lua/diffly/generated.lua` for the full ported list).
- `.gitattributes` `linguist-generated`, as an override in BOTH directions,
  exactly like on github.com: `path linguist-generated` (or `=true`/any
  non-`false` value) forces collapsing even for a file the heuristics would
  otherwise render normally; `path -linguist-generated` (or `=false`) forces
  normal rendering even for a heuristic match, skipping the heuristics
  entirely. A working-tree (uncommitted) `.gitattributes` edit takes effect
  immediately -- it need not be staged or committed. This is the only way to
  configure individual files; there is no separate diffly-specific pattern
  list.

Two rules linguist itself doesn't have, so diffly doesn't add them either: no
generic `@generated`-marker rule, no generic "DO NOT EDIT" rule. `yarn.lock`,
`Gemfile.lock`, and `go.sum` are deliberately NOT treated as generated
(linguist doesn't either) -- opt them in via `.gitattributes` if you want them
collapsed.

Precedence ~
- Binary detection always wins outright (nothing further to load).
- The large-file guard (|diffly-large-files|) runs next: an oversized file's
  content is never loaded, so the generated-file heuristics -- which need to
  read that content -- never get a chance to run for it. A `.gitattributes`
  override never reaches this case either, since the size guard already
  returned.
- Only once neither of the above applies does a `.gitattributes` override or
  (failing that) the content heuristics decide.

Set `collapse_generated = false` to disable heuristic AND
`.gitattributes`-override collapsing entirely.

7. Viewed-state semantics                                 *diffly-viewed-state*

Marking files "viewed" is manual only -- nothing is auto-marked on scroll --
and is shared between the panel, the diff buffers, and real file buffers: `v`
(panel/diff buffers), `<leader>v` (the universal layer, `keymaps.universal`,
which also works on the panel and diff buffers), and
`<Plug>(diffly-toggle-viewed)` all record the same mark.

Key ~
Viewed marks are persisted keyed by:
- `owner/repo#number`, when a GitHub PR for the current branch is detected
  via `gh`; or otherwise
- the repo identity plus the `base..head` branch pair.

These two keyspaces never mix: opening the same branch pair through a PR one
day and without `gh` the next tracks viewed state separately, by design (see
docs/design.md). Repo identity is the normalized remote URL when one exists
(so state is shared across worktrees and clones of the same remote),
otherwise the worktree's toplevel path.

Invalidation ~
Marking a file records the pair of blob SHAs (base side, right-hand side) it
had at that moment. If either side's blob no longer matches at render time --
because either branch moved on -- the file counts as un-viewed again,
GitHub-style. Files nobody touched keep their mark across any number of new
commits on either side.

Persistence ~
State lives under `stdpath('data') .. '/diffly'`, one JSON file per review key
(named after a hash of the key, see `lua/diffly/state.lua`). Use `:Diffly
clean` / `:Diffly clean all` to remove it; there is no automatic
garbage-collection.

8. Bulk viewed marking                                     *diffly-bulk-viewed*

Marking files "viewed" stays manual-trigger-only (see |diffly-viewed-state|
above) -- nothing is ever marked as a side effect of scrolling, opening, or
refreshing a diff. `S`/`:Diffly sweep [{name}]` and `V` are simply two more
explicit triggers, for marking (or unmarking) several files in one step
instead of one `v` press per file -- e.g. lockfiles and generated output
nobody reads line-by-line.

viewed_patterns ~
An ORDERED list of pattern GROUPS, matched against every file currently in
the diff. Each item is either:
- a plain glob STRING -- every such string collects into one implicit group
  named `"default"`, positioned wherever the FIRST string appears in the
  list (this is the pre-groups shape, still fully supported); or
- a TABLE `{ name = "...", patterns = {...} }` for an explicitly named
  group, so unrelated batches (e.g. lockfiles vs. generated output) can be
  swept independently.

Within a group, glob semantics are unchanged from before groups existed:
- A pattern with NO "/" matches the entry's BASENAME, anywhere in the tree --
  `"*.lock"` matches `yarn.lock`, `packages/api/Gemfile.lock`, etc.
- A pattern CONTAINING "/" matches the full toplevel-relative path instead --
  `"dist/**"` matches everything under a top-level `dist/`, but not
  `packages/api/dist/bundle.js`.
- Patterns are compiled with `vim.glob.to_lpeg` (the same glob dialect
  Neovim's LSP client uses): `**` crosses directory boundaries, a single `*`
  does not.

Example: >lua
  require("diffly").setup({
    viewed_patterns = {
      {
        name = "lock files",
        patterns = { "*.lock", "*.sum" }, -- yarn.lock, go.sum, ...
      },
      {
        name = "generated files",
        patterns = { "*.snap", "dist/**", "**/generated/**" },
      },
    },
  })
<
Backward compatible: a flat list of strings still works exactly as before --
`viewed_patterns = { "*.lock", "dist/**" }` -- collapsing into one implicit
`"default"` group, so existing configs need no changes.

An invalid pattern never raises: it is skipped, and warned about once per
Neovim session (not once per sweep). Two groups sharing the same name merge
into the first occurrence's group the same way, also warned once.

Choosing a group ~
`S` / `:Diffly sweep` (with no {name}):
- 0 groups configured -- notifies that `viewed_patterns` isn't set; nothing
  happens.
- exactly 1 group configured -- sweeps it immediately, no prompt.
- 2+ groups configured -- prompts via |vim.ui.select()| (`prompt = "Sweep
  pattern group:"`), so a picker plugin that has replaced Neovim's builtin
  `vim.ui.select` (telescope, fzf-lua, dressing.nvim, ...) applies
  automatically. The first entry is always `all groups (N files, M
  unviewed)` -- the union of every group, a file matched by more than one
  group counted only once -- followed by each configured group as `<name>
  (N files, M unviewed)`. Cancelling out of the prompt changes nothing.

`:Diffly sweep {name}` sweeps that one group directly, skipping the prompt
entirely: it matches an exact name first, then falls back to a UNIQUE
prefix (so `:Diffly sweep lock` works when `"lock files"` is the only
configured group starting with `"lock"`). A name containing spaces works
either typed by hand (`:Diffly sweep lock files`) or via `<Tab>` completion,
which offers the live review's group names with embedded spaces
backslash-escaped (see |diffly-commands|). An unresolved name reports which
groups ARE configured instead of silently doing nothing.

Tri-state toggle ~
A pattern-group sweep (`S`/`:Diffly sweep [{name}]`) and `V` on a directory
row (every file in that subtree) share the same tri-state rule: if at least
one file in the swept scope is currently un-viewed, the WHOLE BATCH is
marked viewed; only once EVERY file in scope is already viewed does
sweeping it again unmark them all. Repeating the same action is therefore a
clean toggle, not a one-way ratchet. `V` on a file row is not a batch at all
-- it behaves exactly like `v`.

Each batch persists with a single save (not one write per file) and reports a
compact result scoped to what was actually swept, e.g. `diffly: marked 5
files as viewed (lock files)` / `diffly: unmarked 5 files (all groups)`.
Progress (`3/12 viewed`) updates exactly like any other mark.

`V` on a directory sees that subtree's FULL file list regardless of the
panel's `H` filter or any folds -- both are display concerns only, same as
everywhere else in the panel (see |diffly-keymaps|).

`config.auto_advance` applies to a batch exactly like `v`: a batch that
MARKED files jumps to the next un-viewed file afterward (`]f`-style,
resolved from the start of file order, since a batch has no single "current"
file to resume from); a batch that UNMARKED files never advances.

9. Comments                                                   *diffly-comments*

difit-style line comments: notes attached to a line (or visual range) on
either side of the diff, written in a small markdown float and rendered
inline right below the code they annotate -- below the DELETED lines, for a
base-side comment on removed code. Each thread is a boxed block: `╭─ ✎ draft`
for local notes, `╭─ @author` for PR review threads -- so several comments on
one line stay visually separate and local drafts are never mistaken for
conversations already on the forge. `<leader>ct` collapses every comment down
to a `✎` end-of-line marker when the inline text gets in the way. The panel
shows a per-file `✎N` count.

Comments are local: they are stored next to the viewed marks in the same
per-review state file (same key, same worktree/clone sharing -- see
|diffly-viewed-state|), whether or not a PR exists. GitHub only ever enters
as a read-only overlay (below) and as `:Diffly submit`'s explicit target.

Submitting a review ~
`:Diffly submit` turns the local drafts into ONE GitHub review. Every draft
is validated against the PR's real (merge-base..head-commit) diff first:
drafts on uncommitted edits, lines outside the diff, ranges crossing hunks,
or outdated anchors are reported and stay local -- GitHub's reviews endpoint
is all-or-nothing, so pre-validation keeps one bad line from failing the
batch. Pick the event (Comment / Approve / Request changes; the forge
rejects approving your own PR), optionally type a summary into the compose
float (<C-s> submits, an empty summary is fine), and one review lands -- a
single notification for your reviewers. Submitted drafts leave the local
store and come back through the overlay, so nothing renders twice. Local
HEAD must be the PR's head commit; push, pull, or `gh pr checkout` first.

Drafts written BEFORE the PR existed are adopted automatically: the first
time the branch's review opens PR-keyed, its branch-pair drafts move into
the PR store (viewed marks deliberately stay behind).

PR review threads (the read-only overlay) ~
When the review is keyed to a GitHub PR, its existing review conversations
are fetched through `gh` -- asynchronously on open (the UI never waits on
the network) and again on any EXPLICIT refresh (the panel's `R`, `:Diffly
refresh`, a bare `:Diffly` on the viewer tab); saving a file never triggers
a fetch. Threads render read-only in the diff like local notes, but
attributed (`@author` lines, replies included) and visually distinct
(|hl-DifflyCommentRemoteMarker|). Unresolved threads show by default;
`<leader>cr`/`cr` reveals resolved ones, session-wide. Outdated threads
(whose code is gone from the PR head) never render inline but appear in
`:Diffly comments` marked `[outdated]`; every remote entry there is tagged
`[@author]`. The panel's `✎N` count includes unresolved remote threads,
independent of the toggle. Thread positions are PR-head coordinates: exact
on a clean `gh pr checkout`, an approximation once local edits shift lines.

Writing ~
`<leader>ca` (any diffly context) / `ca` (diffly-owned buffers) opens the
compose float at the cursor: type the note (markdown), then save with `:w`
or `:wq` (the buffer is 'acwrite') or `<C-s>`; `q` -- or saving an empty
body -- cancels, and `:q` on an unsaved body warns like any unsaved file
(`:q!` discards). Prefer `:w` when `<C-s>` doesn't arrive: terminal flow
control (XON/XOFF) or a multiplexer binding can swallow Ctrl+S before Neovim
sees it. In visual mode the same key comments on the selected line range. Base-side comments (including deleted lines) are
written from the side-by-side LEFT window; the unified view writes head-side
comments only, though it renders both sides.

`<leader>ce`/`ce` edits and `<leader>cd`/`cd` deletes the comment under the
cursor (delete confirms via |vim.ui.select()|; when several comments cover
the same line, both prompt for which one).

Following the code ~
Each comment remembers the exact text it was written against. When the file
changes -- new commits, or an external tool rewriting it mid-review -- the
anchor is re-located by searching for that text near its old position, and
follows it. If the commented lines are gone entirely, the comment turns
OUTDATED: it stops rendering inline (a note pinned to the wrong line is
worse than none), but stays in the panel `✎N` count and in `:Diffly
comments`, where `[outdated]` marks it. `[base]` marks base-side comments,
whose line numbers refer to base content while quickfix jumps into the
worktree file.

AI prompt copy ~
`<leader>cy`/`cy` copies the comment under the cursor, `<leader>cY`/`cY`
every comment in the review, to the unnamed register (and the system
clipboard when a provider is available) in difit's prompt format -- ready to
paste into an AI coding agent: >
  src/api/client.ts:L42-L48
  this retry loop needs a backoff

  =====

  src/api/client.ts:L102
  copy the error message into the exception
<

==============================================================================
10. GitHub integration                                          *diffly-github*

The `gh` CLI is entirely optional. When it is missing, not authenticated, or
the current branch has no associated pull request, diffly.nvim silently falls
back to the branch-pair key described in |diffly-viewed-state| -- nothing
about opening or reviewing a diff depends on `gh` being present.

When `gh` *is* available and a PR is detected, it is used for two independent
things: as a fallback source for the base branch (still beaten by `:Diffly
{base}` or `config.base`), and as the viewed-state key. The diff itself is
always generated from local git against your current worktree/HEAD --
reviewing someone else's PR still requires checking it out locally first
(e.g. via `gh pr checkout`).

11. Agent bridge                                                 *diffly-agent*

`bin/diffly` (in the plugin directory) lets coding agents read and write the
same review comments you see in the editor: >
  diffly info                                # review metadata as JSON
  diffly comments list [--remote]            # draft (and PR) threads
  diffly comments add --file {path} --line {n} [--end-line {m}]
                      [--side base|head] --body {text}|- [--author {name}]
  diffly comments reply {id} --body {text}|-
  diffly comments rm {id}
  diffly navigate --file {path} [--line {n}]
  diffly skill install [--dir {path}]
<
Run it from inside the repository under review. Output is JSON on stdout,
errors on stderr; exit codes: 0 ok, 1 error, 2 `navigate` without a live
editor.

Live first ~
The CLI finds a running Neovim that holds this repository's review -- an
explicit `--server {addr}`, the hosting terminal's `$NVIM`, or a scan of
live instances -- and performs every operation inside it over RPC: writes
render in the open UI immediately and never race the editor's own saves.
Without a live session it operates on the persisted review state through
the same plugin code; `--headless` forces that path. A Ctrl-Z-suspended
editor still answers: the Neovim core runs as a separate server process,
only the TUI sleeps.

Agent comments are normal drafts ~
Messages written through the CLI carry an author (default "agent"): they
render as `✎ draft @agent`, count in the panel's `✎N`, are tagged
`[@agent]` in `:Diffly comments`, re-anchor when the code moves, and submit
with the rest of your review (|diffly-comments|). Deliberately not exposed
to agents: marking files viewed, and `:Diffly submit`.

Agent skill ~
`diffly skill install` writes a ready-made Claude Code skill to
`~/.claude/skills/diffly-review/` (or `--dir {path}`, e.g. a project's
`.claude/skills`), with this checkout's absolute `bin/diffly` path baked
in. Reinstall after moving the plugin directory; the file is a generated
artifact and is simply overwritten.
